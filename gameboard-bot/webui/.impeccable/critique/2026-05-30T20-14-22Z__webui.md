## Anti-Patterns Verdict

**LLM assessment**: Cleaner than 90% of comparables. No gradient text, no glassmorphism cards, no hero-metric template, no infinite identical icon grids, no heavy gamification chrome. The `island-shell` uses `border + box-shadow` but the shadow blur is 12px (below the 16px ghost-card threshold). Border-radius on the main card is 28px (right at the spec ceiling). The `island-kicker` eyebrow pattern appears in multiple sections per page but stays defensible — each kicker sits under its own named section header. The numbered rule cards in HowToPlay (01–04) are the most borderline tell: rules ARE a genuine sequence, but the `island-kicker` treatment on every card in a 2×2 grid reads very close to the banned numbered-eyebrow scaffolding.

The CSS variable names (`--sea-ink`, `--lagoon`, `--foam`) are coastal-era artifacts from a previous palette. Users don't see them, but they signal a doc-rot drift worth addressing.

**Deterministic scan**: Detector returned exit 0, empty findings. No automated pattern violations across `src/routes` and `src/components`.

**Browser visualization**: Not attempted — browser automation unavailable in this session.

## Overall Impression

This UI punches above its weight. The visual identity is coherent and deliberate — the palette restraint is impressive in practice, not just in principle. The game experience is genuinely board-first. The biggest gap: the interface assumes players already know this game. First-timers can open "How to Play," but the game itself offers no on-ramp — no contextual cues during play, no shortcut discoverability, no warning before accidentally exiting a game in progress.

## What's Working

1. **60-30-10 execution**: The white/ink/brown ratio holds across every screen. Brown appears sparingly and earns every pixel. The board dominates without competing chrome.
2. **Accessibility foundations**: `aria-label` on every board cell, `aria-live="polite"` on turn status, `role="dialog" aria-modal="true"` on the modal, `aria-pressed` on toggle buttons. The structural groundwork is solid for a game UI.
3. **Board-first layout**: The three-column grid with player cards flanking keeps the board central on desktop. The `order-first` trick pushes the board to the top of the mobile stack — correct priority.

## Priority Issues

**[P1] Keyboard shortcuts invisible: zero discoverability**
- **What**: M/W/H/V/R/? shortcuts are implemented in game.tsx but not surfaced anywhere in the UI. HowToPlay modal doesn't mention them. The `?` key opens help, but help doesn't list shortcuts.
- **Why it matters**: The developer/bot-curious audience is explicitly named as a primary user type. They'd use and expect shortcuts. First-timers will never find them. A feature that can't be found is a feature that doesn't exist.
- **Fix**: Add a keyboard legend to the HowToPlay modal. At minimum, a small hint strip near the action controls: "M=move  W=wall  H/V=orient  R=reset  ?=help."
- **Suggested command**: `$impeccable harden`

**[P1] No mid-game navigation guard: state lost silently**
- **What**: Clicking "← Change mode" or any header nav link during an active game exits immediately with no confirmation. Game state is component-local (not persisted), so navigating away loses it.
- **Why it matters**: A player mid-game who accidentally taps a nav link loses their game. No `beforeunload` warning, no confirmation prompt. Especially risky on mobile where nav targets are closer together.
- **Fix**: Detect when `turns > 0` and the user navigates away. Show a brief confirm prompt ("Leave this game? Progress will be lost.") or persist state to `sessionStorage`.
- **Suggested command**: `$impeccable harden`

**[P2] Double kicker in the game's board column**
- **What**: `island-kicker` appears twice in the same vertical column: `{status}` (SOUTH to move / Bot thinking…) above the board, and "Win chance" inside WinMeter — both uppercase tracked labels in quick succession.
- **Why it matters**: DESIGN.md specifies "one per section maximum." Two kickers in the same visual column dilute each other. The status is primary; "Win chance" is redundant noise (the meter is self-explanatory with its percentage labels and colored bar).
- **Fix**: Remove `island-kicker` from WinMeter's "Win chance" label. Use `text-xs font-semibold text-[var(--sea-ink-soft)]` or drop the label entirely.
- **Suggested command**: `$impeccable polish`

**[P2] HowToPlay 01–04 numbered kickers: close to the banned pattern**
- **What**: Each rule card uses `island-kicker` for `01`, `02`, `03`, `04`. A 2×2 card grid with a numbered kicker on every card is exactly the "numbered section eyebrow" pattern the skill bans.
- **Why it matters**: The numbers don't carry information the reader needs — rules can be read in any order. Removing them loses nothing and removes the reflex-scaffolding tell.
- **Fix**: Drop the number from the kicker style. Use plain `text-sm font-bold text-[var(--sea-ink-soft)]` for the step number, or remove numbers and let the rule titles stand alone.
- **Suggested command**: `$impeccable polish`

**[P2] Graph Traveler interaction hints are sub-pixel-visible**
- **What**: "drag node · drag bg to pan · wheel to zoom · click to fly" is `text-xs text-[var(--sea-ink-soft)]` inline with the section kicker. On a dense screen it disappears.
- **Why it matters**: The graph requires non-obvious interactions that can't be guessed. The developer audience expects discoverability. Missing the hint means users stare at a static graph.
- **Fix**: Move the hint text below the graph header on its own line, or surface it as a brief icon row (drag icon, scroll icon, click icon) with labels. Minimum: bump text to `text-[var(--sea-ink)]` so it's visible.
- **Suggested command**: `$impeccable clarify`

## Persona Red Flags

**Alex (Power User / Developer)**:
- Keyboard shortcuts M/W/H/V/R/? exist but are invisible from the UI. Alex will check HowToPlay, find nothing, and move on — never knowing.
- Graph `move → (r,c)` notation is appropriate for Alex but no legend is provided.
- No way to step back a move (intentional game design, but power users will try Ctrl+Z and be surprised).

**Jordan (First-Timer)**:
- Goal rows have a warm-brown tint (`board-goal`) but no label. Jordan may not connect the subtle tint to "where I need to reach."
- After placing a wall, action mode silently resets to "move." Jordan won't know why "Wall" is no longer selected.
- "SOUTH starts on row 1, NORTH on row 9" in the rules uses game-internal labels that aren't shown on the board itself. Jordan plays without knowing which row is 1 and which is 9.

**Sam (Accessibility-Dependent)**:
- Tab order in the game's three-column grid follows DOM order (North card → board → South card) but CSS visual order puts the board first (`order-first`). A keyboard user tabs through North card before reaching the board even though they see the board first — spatial mismatch.
- HowToPlay dialog has `aria-label` but no `aria-describedby` — screen reader won't auto-read the intro paragraph on dialog open.
- Wall placement hit-strip buttons only render in wall mode. Screen reader users may not receive notification when this layer appears/disappears.

## Minor Observations

- PRODUCT.md Design Principle 3 says "the teal/sea palette is the brand" — but the palette has no teal. This is a doc inconsistency from a previous palette pivot, not a visual problem.
- CSS variable names (`--sea-ink`, `--lagoon`, `--foam`) are coastal-era artifacts. Users don't see them but they'll confuse future contributors.
- About page uses `text-[var(--sea-ink-soft)]` for ALL body paragraphs — including the lead paragraph. With nothing in ink to contrast against, the soft treatment makes all copy feel secondary. The first sentence at least should be ink.
- `html lang="en"` ✓, viewport meta ✓, reduced-motion `@media` ✓, `selection:bg` on-brand ✓.
- Win state uses `🎉` emoji — the only decorative element. Borderline against the restraint principle, but tonally appropriate.
- Footer is minimal and correct. `new Date().getFullYear()` in SSR context is fine.

## Questions to Consider

- "What if the goal rows showed a one-word label ('Goal') on first load, fading after the first move? First-timers get the orientation; experienced players ignore it."
- "The Graph Traveler is accessible only through the header nav. Would it land better as a drawer or secondary panel from the game screen, where the developer audience is already present?"
- "Both the board and the WinMeter dim when the bot is thinking. Two dimming elements stacked may read as the UI failing rather than 'bot is working.' What if only one dimmed?"
